# ADR-004 — Workflow миграций БД (Alembic в репозитории)

| | |
|---|---|
| **Статус** | Accepted |
| **Дата** | 2026-04-20 |
| **Автор** | Команда проекта |

---

## Контекст

[ADR-001](adr-001-database.md) фиксирует **PostgreSQL**; [ADR-003](adr-003-orm.md) — **SQLAlchemy 2 async**, **asyncpg** и **Alembic** для эволюции схемы. До принятия этого ADR не были задокументированы: расположение каталогов в `backend/`, соглашения по ревизиям, использование autogenerate, политика downgrade и связь `alembic.ini` / `env.py` с URL базы и метаданными ORM.

Цель — один воспроизводимый способ менять схему в разработке и CI без споров «куда класть миграции» и «можно ли править файл ревизии руками».

---

## Рассмотренные варианты

### 1. Alembic в корне монорепозитория
**Плюсы:** одна точка входа, если бы несколько сервисов делили БД.  
**Минусы:** сейчас единственный потребитель схемы — `backend/`; лишняя вложенность путей для `uv run`.

### 2. Alembic внутри `backend/` (выбранная модель)
**Плюсы:** зависимости и lock уже в `backend/pyproject.toml`; команды `uv run alembic` без смены контекста; соответствие структуре из vision (`storage`, будущий пакет ORM).  
**Минусы:** при появлении второго сервиса с той же БД потребуется явная политика (отдельный ADR).

### 3. Только ручной SQL без autogenerate
**Плюсы:** полный контроль.  
**Минусы:** больше расхождений с моделями SQLAlchemy; медленнее итерация. Допустимо точечно, но не как единственный режим.

---

## Решение

1. **Корень инструмента миграций** — каталог **`backend/`**:
   - `backend/alembic.ini` — конфигурация Alembic;
   - `backend/alembic/` — `env.py`, `versions/`, шаблон скрипта ревизии;
   - файлы ревизий — **`backend/alembic/versions/*.py`**.

2. **Имя ревизии (slug в сообщении):** короткая английская фраза в **snake_case** или kebab-case в `-m`, отражающая смысл (`add_users_table`, `add_appointments_fk`). Префикс ревизии (hex id) задаёт Alembic автоматически.

3. **Источник правды для autogenerate:** единый объект **`MetaData`** SQLAlchemy (или `DeclarativeBase`), собирающий все mapped classes приложения. Ревизии с **autogenerate** создаются после изменения моделей и ручного просмотра диффа.

4. **Ручные правки в файле ревизии:** допустимы для данных, частичных индексов, `CREATE EXTENSION`, правок невидимых autogenerate; обязательно **комментарий вверху файла** или в теле `upgrade()` о причине.

5. **Downgrade:** каждая ревизия **должна** иметь осмысленный `downgrade()` для отката в dev; перед релизом миграций в общую ветку — прогон `upgrade → downgrade → upgrade` на чистой БД. В production откат только по согласованной процедуре (не «тихий» downgrade без бэкапа).

6. **Подключение к БД:** URL из переменной окружения **`DATABASE_URL`** (или эквивалент, зафиксированный в `backend` config). Формат для приложения: `postgresql+asyncpg://...`. Для Alembic в типовом **синхронном** `env.py` допускается отдельный URL с драйвером `psycopg` (v3) или преобразование DSN на `postgresql+psycopg://` — конкретная строка подключения задаётся при реализации **iter-db-04/05**, пока достаточно требования «один DSN из env».

7. **Связь с ADR-001/003:** целевая СУБД — PostgreSQL; ORM и Alembic не заменяют контракты API (Pydantic в роутерах, маппинг в репозиториях).

---

## Обоснование

Концентрация Alembic в `backend/` снижает трение для разработчиков, уже работающих в этом пакете. Autogenerate ускоряет согласование с [data-model.md](../data-model.md), а явная политика downgrade снижает риск необратимых миграций в общих ветках.
