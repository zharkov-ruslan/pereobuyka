## Контекст

СУБД зафиксирована в [ADR-001](adr-001-database.md): **PostgreSQL** в production. В `docs/vision.md` слой **storage / репозитории** изолирует работу с БД; домен описан в `docs/tech/data-model.md` как реляционный (связи, транзакции).

HTTP-слой в [ADR-002](adr-002-backend-framework.md) использует **Pydantic** для контрактов API — это не замена слоя персистентности: нужно явно решить, **как** backend выполняет запросы к PostgreSQL и как ведётся эволюция схемы.

Требования:
- асинхронный доступ к БД в одном стиле с FastAPI (без блокировки event loop в типовых путях);
- выразительная работа с связями и транзакциями;
- воспроизводимые **миграции** схемы (версионируемый SQL или эквивалент);
- согласование с экосистемой Python 3.12+ и упоминанием SQLAlchemy в обосновании ADR-001.

---

## Рассмотренные варианты

### 1. SQLAlchemy 2.x (ORM + Core), async, драйвер asyncpg
**Плюсы:** де-факто стандарт в Python для реляционных БД; единая модель от Core до ORM; нативный async-режим в 2.x; хорошая интеграция с **Alembic** для миграций; обширные гайды и интеграция с pytest (транзакционные фикстуры и т.д.).  
**Минусы:** кривая обучения 2.x; нужна дисциплина, чтобы не тащить ORM-модели в API-слой.

### 2. SQLModel
**Плюсы:** объединяет Pydantic и SQLAlchemy, меньше дублирования типов при узком совпадении API и таблиц.  
**Минусы:** надстройка над SQLAlchemy — привязка к релизам и меньшая гибкость, чем у «чистого» SQLAlchemy; для разведения DTO API и таблиц всё равно часто появляются отдельные модели. Подходит как вариант точечно, не как обязательная база всего проекта.

### 3. Tortoise ORM
**Плюсы:** asyncio-first API.  
**Минусы:** экосистема уже слабее SQLAlchemy; дублирование знаний команды, если везде ожидается SQLAlchemy/Alembic.

### 4. Peewee
**Плюсы:** простота для небольших проектов.  
**Минусы:** слабее по async и зрелости для PostgreSQL+масштаб по сравнению с SQLAlchemy в данном домене.

### 5. Только asyncpg (или psycopg) + ручной SQL
**Плюсы:** полный контроль и предсказуемые запросы.  
**Минусы:** для множества сущностей и связей — много шаблонного кода и риск ошибок; миграции всё равно нужны отдельным инструментом. Имеет смысл для узких горячих путей, не как единственный слой всего backend.

---

## Решение

- **ORM:** **SQLAlchemy 2.x** в **асинхронном** режиме работы с БД; драйвер для PostgreSQL — **asyncpg** (через соответствующий Dialect SQLAlchemy).
- **Миграции схемы:** **Alembic**, опирающийся на метаданные/модели SQLAlchemy.

**Граница слоёв:** схемы **Pydantic** — вход/выход HTTP API; **mapped classes / таблицы** SQLAlchemy — персистентность. Преобразование между ними — в **storage / репозиториях** (или в сервисах при явной договорённости), без смешения ORM-моделей с ответами API в роутерах. Исключения (например, общий тип для простой сущности) допустимы локально и осознанно.

---

## Обоснование

Решение согласуется с ADR-001 (экосистема SQLAlchemy) и с архитектурой vision: репозитории остаются местом инкапсуляции доступа к данным, при этом ORM снимает рутину по связям и транзакциям. Alembic — зрелый стандарт миграций для SQLAlchemy и закрывает открытый пункт в ADR-001 про выбор инструмента миграций.

---

## Последствия

- Зависимости (`sqlalchemy[asyncio]`, `asyncpg`, `alembic`, при необходимости `pydantic-settings`) фиксируются в `pyproject.toml` каталога `backend/` при появлении каркаса (см. tasklist, iter-03).
- Локальная разработка: поднятый PostgreSQL и применённые миграции — часть воспроизводимого старта; SQLite остаётся опциональным только для dev по ADR-001, без смены целевого стека ORM.
- Тесты: при интеграционных тестах с БД — транзакции/откаты или отдельная тестовая БД; детали выносятся в реализацию, не в этот ADR.
