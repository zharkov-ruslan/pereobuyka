# ADR-001 — Выбор СУБД

| | |
|---|---|
| **Статус** | Accepted |
| **Дата** | 2026-04-13 |
| **Автор** | Команда проекта |

---

## Контекст

Система «Переобуйка» — мультиканальная: Telegram-бот и веб-приложение работают как клиенты единого backend. Доступ к БД — только через backend; бот и веб обращаются к данным исключительно через его API. Модель данных реляционная: пользователи, услуги, записи, визиты, бонусы — со строгими связями и транзакционными требованиями (например, фиксация визита + начисление бонусов должны быть атомарны).

Требования к СУБД:
- поддержка реляционной модели с FK и транзакциями;
- конкурентный доступ (несколько процессов/воркеров backend);
- возможность расширения при росте нагрузки;
- простой деплой и наличие managed-вариантов.

---

## Рассмотренные варианты

### 1. SQLite
**Плюсы:** нет отдельного процесса, нулевая инфраструктура, идеален для локальной разработки.  
**Минусы:** не поддерживает конкурентную запись из нескольких процессов; непригоден для деплоя с отдельным backend-сервисом. При любом горизонтальном масштабировании — тупик.

### 2. PostgreSQL
**Плюсы:** зрелая реляционная СУБД; строгие транзакции (ACID); конкурентный доступ; JSONB для гибких данных; встроенный полнотекстовый поиск; расширение pgvector при появлении семантического поиска; богатая экосистема (asyncpg, psycopg3, SQLAlchemy); доступны managed-варианты (Yandex Cloud, Supabase, Railway).  
**Минусы:** требует отдельного процесса/сервиса; чуть сложнее в локальном старте (решается Docker Compose).

### 3. MySQL / MariaDB
**Плюсы:** широко распространён, есть managed-варианты.  
**Минусы:** исторически слабее PostgreSQL в поддержке сложных типов, оконных функций и расширяемости. Нет очевидных преимуществ перед PostgreSQL для данного стека.

### 4. MongoDB (документная БД)
**Плюсы:** гибкая схема, горизонтальное масштабирование.  
**Минусы:** модель данных проекта строго реляционная — связи между сущностями важны, транзакции между коллекциями сложнее. Избыточно и не подходит по природе данных.

---

## Решение

**PostgreSQL** — основная СУБД проекта с первого деплоя.  
**SQLite** — только для локальной разработки без инфраструктуры (опционально, по усмотрению разработчика).

---

## Обоснование

PostgreSQL закрывает все текущие и прогнозируемые требования без избыточности:
- реляционная модель с транзакциями — первый класс;
- конкурентный доступ из нескольких процессов backend — штатный режим работы;
- pgvector при необходимости семантического поиска по FAQ — расширение, а не замена СУБД;
- стек Python (asyncpg / SQLAlchemy) — зрелый и хорошо документированный;
- managed-варианты снижают операционную нагрузку при деплое.

---

## Последствия

- Локальная разработка требует запущенного PostgreSQL (рекомендуется через Docker Compose).
- Миграции схемы: инструмент и связка с ORM зафиксированы в [ADR-003](adr-003-orm.md) (**Alembic** + SQLAlchemy).
- SQLite не используется в production и не тестируется в CI как целевая СУБД.
