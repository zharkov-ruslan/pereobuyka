# ADR-002 — Выбор HTTP-фреймворка backend

| | |
|---|---|
| **Статус** | Accepted |
| **Дата** | 2026-04-17 |
| **Автор** | Команда проекта |

---

## Контекст

Backend «Переобуйки» — единое ядро для бота и веб-клиентов: REST API, строгая валидация входа/выхода, машиночитаемый контракт для клиентов и агентов разработки. Стек проекта: Python 3.12+, управление зависимостями через **uv**, целевая структура репозитория в `docs/vision.md`.

Требования к фреймворку:
- асинхронный I/O (запросы к БД и внешним API без блокировки event loop);
- **OpenAPI** из кода (схемы запросов/ответов и `/docs` без ручного дублирования);
- валидация и сериализация, согласованная с экосистемой Python (желательно **Pydantic**);
- предсказуемая интеграция с DI и тестированием (pytest, httpx).

---

## Рассмотренные варианты

### 1. FastAPI
**Плюсы:** зрелая экосистема; OpenAPI и Swagger UI «из коробки»; нативная связка с Pydantic v2; `Depends()` для явного внедрения конфигурации, сервисов и репозиториев; асинхронные эндпоинты; обширная документация и примеры под PostgreSQL/SQLAlchemy.  
**Минусы:** магия аннотаций и стека зависимостей требует дисциплины (тонкие роутеры, логика в сервисах).

### 2. Litestar
**Плюсы:** современный async-фреймворк, встроенные валидация и OpenAPI, активное развитие.  
**Минусы:** меньше готовых рецептов и библиотечных интеграций по сравнению с FastAPI; для команды выше стоимость поиска решений на нестандартных задачах.

### 3. Django + Django REST Framework
**Плюсы:** батарейки, админка, ORM из одной экосистемы.  
**Минусы:** тяжелее для узкого API-ядра без админки на первом этапе; async-история сложнее и менее идиоматична; избыточность относительно принципов KISS/YAGNI в `docs/vision.md`.

### 4. Starlette «голый»
**Плюсы:** минимализм и полный контроль.  
**Минусы:** OpenAPI и схемы пришлось бы собирать вручную или прикручивать отдельно — дублирование и риск расхождения с кодом.

### 5. Flask / Quart
**Плюсы:** простота для маленьких сервисов.  
**Минусы:** async-сторона у Quart менее унифицирована в типовых гайдах; OpenAPI не является стандартом де-факто для стека.

---

## Решение

**FastAPI** — основной HTTP-фреймворк backend-сервиса.

---

## Обоснование

- Закрывает требование контрактной документации (OpenAPI) без отдельного слоя поддержки.
- Pydantic совпадает с ожидаемым стеком валидации и с последующими схемами API в коде.
- `Depends` позволяет держать роутеры тонкими и направлять работу в `services/` и `storage/`, в духе `docs/vision.md`.
- Наибольшая предсказуемость интеграций (тест-клиенты, middleware, lifespan) для команды и AI-агентов.

---

## Последствия

- Зависимости backend: `fastapi`, ASGI-сервер для запуска (например `uvicorn`), при необходимости — `pydantic-settings` для конфигурации; конкретные версии фиксируются в `pyproject.toml` при появлении каталога `backend/`.
- Версионирование публичного API (`/api/v1/...`) и детали роутинга — в следующих итерациях (контракт и каркас), не в этом ADR.
