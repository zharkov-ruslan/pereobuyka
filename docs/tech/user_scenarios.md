# Требования к данным по сценариям (клиент и администратор)

Продуктово-ориентированное описание: **что пользователь видит и что может сделать**, и **какие данные и связи** нужны в системе для базовых сценариев. Без привязки к SQL и физической схеме — детали таблиц и типов уточняются в [data-model.md](data-model.md) и на этапе физического проектирования (см. [tasklist-database.md](../tasks/tasklist-database.md)).

**Опора:** сценарии из [vision.md](../vision.md) §4 · концептуальные сущности — [data-model.md](data-model.md).

---

## 1. Каналы: бот и веб

| Аспект | Что общее | Что различается по каналу |
|--------|-----------|---------------------------|
| Учёт пользователя | Один профиль `User` в БД независимо от канала | В боте при первом контакте известен `telegram_id`; в вебе — регистрация без Telegram (идентификация по телефону, метод уточняется в продукте) |
| Запись, прайс, бонусы | Одни и те же сущности и бизнес-правила backend | Отличается только UI/UX: экраны веба vs диалоги и кнопки в Telegram |
| Уведомления | Факт визита/бонусов хранится в данных | Push в Telegram — канал доставки (см. интеграции), не отдельная сущность «уведомление» в MVP |

Все каналы ходят в **один backend**; в БД не дублируются «копии» записей под канал — достаточно связи записи с `user_id` и при необходимости полей происхождения в `User` (`source`).

## 2. Матрица «сценарий vision → сущности и фокус данных»

Нумерация сценариев клиента и администратора соответствует таблицам в [vision.md](../vision.md) §4.

### Клиент

| # | Сценарий (vision) | Что видит клиент | Что делает | Сущности и минимальный состав данных |
|---|---------------------|------------------|------------|--------------------------------------|
| 1 | Регистрация / профиль | Имя, телефон, при наличии — привязка к Telegram | Первичное создание профиля или вход | `User`: идентификатор, `name`, `phone`, `role` (client), `telegram_id` или null, `registered_at`, `source` |
| 2 | Консультация (LLM) | Ответы на вопросы об услугах, ценах, слотах | Задаёт вопросы в свободной форме | Для **фактов** — те же данные, что в сценариях 3–4 и 7 (чтение `Service`, слотов, бонусов); для **шаблонных ответов** — `FAQ` (`question`, `answer`, `is_active`). Сама переписка в БД в MVP не обязательна |
| 3 | Услуги и цены | Список активных услуг: название, описание, цена, длительность | Просматривает каталог | `Service`: `name`, `description`, `price`, `duration_minutes`, `is_active` |
| 4 | Запись | Свободные окна на дату (начало/конец), итог по времени и сумме | Выбирает услуги, слот, подтверждает | `ScheduleRule` / `ScheduleException` (правила доступности), существующие `Appointment` (занятость), новая `Appointment` + связь с услугами (`AppointmentService`): `starts_at`, `ends_at`, `total_price`, `status`; список идентификаторов услуг |
| 5 | Просмотр / отмена записей | Список своих записей: дата, время, услуги, статус, сумма | Отменяет при необходимости | `Appointment` по `user_id`, строки услуг записи, смена `status` на отмену при отмене |
| 6 | После визита | История: что было оказано, суммы, бонусы | Просматривает историю (уведомление — канал) | `Visit` (привязка к записи), `Visit`↔услуги при отличии от записи, суммы и бонусы по визиту; связь с `Appointment` |
| 7 | Лояльность | Текущий баланс бонусов; история движений (чтение) | Использует бонусы при оплате/записи по правилам продукта | `BonusAccount` (`user_id`, `balance`), `BonusTransaction` (`type`, `amount`, `visit_id`, `created_at`, при `adjust` — `comment`) |

### Администратор

| # | Сценарий (vision) | Что видит администратор | Что делает | Сущности и минимальный состав данных |
|---|---------------------|-------------------------|------------|--------------------------------------|
| 1 | Прайс | Таблица услуг с ценой, длительностью, активностью | Создаёт / редактирует / деактивирует услуги | `Service` — поля как у клиента; для админки нужны все записи, включая неактивные |
| 2 | Расписание | Шаблон по дням недели; исключения на конкретную дату | Редактирует правила и исключения | `ScheduleRule` (`weekday`, `start_time`, `end_time`, `is_day_off`); `ScheduleException` (`date`, `start_time`, `end_time`, `is_day_off`) |
| 3 | Журнал записей | Список записей с фильтрами (дата, статус и т.д.) | Просматривает загрузку, детали записи | `Appointment` + клиент (`User`), услуги записи, при необходимости связь с будущим `Visit` |
| 4 | Подтверждение визита | Данные записи; фактические услуги; итог к оплате; бонусы | Закрывает визит, фиксирует факт услуг и начисления/списание бонусов | `Visit` (`appointment_id`, `total_amount`, `bonus_spent`, `bonus_earned`, `confirmed_at`, `confirmed_by` → `User` с ролью admin), `Visit`↔услуги; порождение операций по `BonusAccount` / `BonusTransaction` |
| 5 | Бонусы | Счёт клиента, история транзакций | Просматривает; при расхождении — ручная корректировка с комментарием | `BonusAccount`, `BonusTransaction` с типом `adjust` и `comment` |

## 3. Сквозные сценарии (кратко)

Ниже — объединённые «цепочки», которые чаще всего проверяют при проектировании UI и API.

### Регистрация и профиль

- **Клиент:** впервые заходит из бота или веба — в системе появляется или обновляется `User`.
- **Данные:** см. строку «Регистрация» в матрице клиента.

### Каталог и цены

- **Клиент:** видит только услуги с `is_active`, чтобы выбрать набор для записи.
- **Админ:** видит полный прайс и может менять любую позицию.

### Запись (слоты + услуги)

- **Клиент:** выбирает дату и услуги → backend считает длительность и стоимость → показывает свободные окна → клиент подтверждает слот.
- **Данные:** правила/исключения расписания (`ScheduleRule`, `ScheduleException`) + занятость (`Appointment`) + выбранные `Service` → новая `Appointment` и связи «запись–услуга».

### Список и отмена записей

- **Клиент:** список своих предстоящих (и при необходимости прошлых) записей; действие «отменить» для допустимых статусов.
- **Данные:** выборка `Appointment` по пользователю, обновление `status`.

### Подтверждение визита администратором

- **Админ:** по записи фиксирует факт работ, сумму, списание/начисление бонусов.
- **Данные:** новый `Visit`, фактические услуги визита, транзакции бонусов, обновление баланса.

### Баланс и история бонусов

- **Клиент:** видит текущий баланс и историю (начисления, списания) в read-only.
- **Админ:** тот же счёт клиента + возможность ручной корректировки.
- **Данные:** `BonusAccount`, список `BonusTransaction` по счёту.

## 4. Открытые вопросы (не блокируют iter-db-01)

- Детальный метод веб-авторизации по телефону (код SMS, сторонний провайдер) — влияет на поля `User` и поток регистрации, не на перечень сущностей.
- Нужно ли хранить историю диалогов LLM — вне минимального набора для сценария «консультация» на уровне данных продукту достаточно чтения FAQ и фактов из backend.
