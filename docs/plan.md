# Дорожная карта «Переобуйка»

## Организация работ

Этот документ фиксирует этапы развития системы, их цели и критерии завершения (DoD).
Каждый этап даёт понятный рабочий инкремент — продуктовый или платформенный.
Текущий статус этапа отражается явно и обновляется по мере продвижения.

Детализация реализации по областям вынесена в tasklist-документы в `docs/tasks/`.
Каждый tasklist соответствует одной области (backend, database, integrations, frontend, devops)
и конкретному этапу или группе этапов.

## Обзор

«Переобуйка» — комплексная система учёта и записи для шиномонтажного сервиса.
Первый канал для клиента — Telegram-бот с LLM-консультантом; в перспективе добавляются
веб-интерфейс клиента и административная панель. Единый backend обслуживает все каналы.

Развитие идёт от ядра к периферии: сначала фиксируются доменная модель и API-контракты,
затем строятся каналы взаимодействия — бот, затем веб. Первые три этапа строго последовательны, каждый следующий опирается на артефакты предыдущего. Начиная с этапа 3, часть работ по интеграциям и фронтенду может идти параллельно после фиксации контрактов backend.

### Этап 1 — Фундамент: backend и модель данных

**Цель:** Создать ядро системы — доменную модель, слой хранения и API-контракты, на которые опираются все последующие этапы.

**Ценность этапа:**
Без стабильного ядра нельзя строить ни бот, ни веб-интерфейс. Этот этап закладывает единую точку правды для всей бизнес-логики: расчёт слотов, прайс, бонусы, записи. Команда получает чёткие контракты и структуру репозитория, что сокращает переделки на следующих этапах.

**Результат:** Запущенный backend с REST API, миграциями БД, базовым набором сервисов и покрытием ключевых сценариев тестами.

**Статус:** ✅ done

**Факт реализации:** по [tasklist-backend.md](tasks/tasklist-backend.md) выполнены итерации **iter-01–iter-07**; по контракту OpenAPI реализованы маршруты **клиента и администратора** (в т.ч. auth Telegram, записи, визиты, лояльность, CRUD услуг и расписания, подтверждение визита и бонусы). Режим **PostgreSQL** — полный набор; без БД (SQLite) остаются только ранние базовые эндпоинты каталога/слотов/записи.

По [tasklist-database.md](tasks/tasklist-database.md) **область database закрыта** (**iter-db-01–iter-db-05**): миграции Alembic, seed (включая пользователя-админа для `ADMIN_ACTOR_USER_ID`), ORM и интеграционные тесты с Testcontainers.

**Активные области:** нет (этап 1 закрыт относительно текущего DoD в этом документе).

**Что должно появиться:**
- Инициализирован проект (`pyproject.toml`, `uv`, структура репозитория по `docs/vision.md`)
- Реализованы доменные модели и репозитории для всех сущностей из `docs/tech/data-model.md`
- Реализованы сервисы: управление прайсом, расписанием, расчёт слотов, записи, визиты, лояльность
- Задан и описан API-контракт backend (маршруты, схемы запросов/ответов)
- Настроены миграции БД (SQLite для разработки, PostgreSQL-совместимые)

**Что можно проверить:**
- Создание записи через API с корректным расчётом длительности и стоимости
- Расчёт свободных слотов с учётом рабочего расписания и существующих записей
- Начисление и списание бонусов после подтверждения визита

**Выходные артефакты:**
- `backend/` — структура проекта, сервисы, репозитории, модели
- `Makefile` с командами запуска, миграций, тестов
- Миграции БД (`backend/alembic/`)
- `docs/tasks/tasklist-backend.md`, `docs/tasks/tasklist-database.md`
- `docs/tech/user_scenarios.md` (сценарии и данные; iter-db-01)
- актуализация `docs/tech/data-model.md` (логика + физика + ER; iter-db-02)
- `docs/tech/adr/adr-004-database-migrations-workflow.md`, `docs/tech/database-migrations.md` (iter-db-03)
- `docker-compose.yml`, цели `db-*` в `Makefile`, seed-скрипт (iter-db-04)
- `backend/src/pereobuyka/db/`, `storage/postgres_repos.py`, `storage/repositories/`, `api/v1/endpoints/`, интеграционные тесты с Testcontainers (iter-db-05; полнота API этапа 1 — см. tasklist-backend iter-05)

### Этап 2 — Telegram-бот

**Цель:** Подключить первый клиентский канал — Telegram-бот, реализующий базовые сценарии: регистрацию, просмотр прайса, запись, историю визитов, баланс бонусов.

**Ценность этапа:**
Бот — первый канал, через который реальный клиент взаимодействует с системой. После этого этапа продукт уже можно показывать и тестировать с живыми пользователями. Консультации через LLM на этом этапе отсутствуют — только структурированные сценарии; это осознанное ограничение для быстрой проверки ядра.

**Результат:** Работающий Telegram-бот, подключённый к backend: клиент может зарегистрироваться, посмотреть прайс, записаться и узнать баланс бонусов.

**Статус:** ✅ done

**Факт реализации:** проект бота в [`bot/`](../bot/) (aiogram 3.x), **HTTP-клиент** к backend и все целевые сценарии этапа 2: регистрация (`/start` через `GET /api/v1/me` + fallback `POST /api/v1/auth/telegram`), каталог услуг (`/services`), запись (`/book`, FSM), просмотр/отмена записей (`/appointments`), бонусы (`/bonus`), история визитов (`/visits`). Временная схема авторизации бота через **BOT_SECRET** сохранена как интеграционный компромисс текущей версии. Детализация вынесена в [`tasklist-02-bot.md`](tasks/tasklist-02-bot.md) и [`tasklist-02-integrations.md`](tasks/tasklist-02-integrations.md).

**Активные области:** backend, integrations

**Зависимость:** API-контракты и backend доступны (см. этап 1); полное завершение этапа 1 по БД/домену не блокирует текущий срез бота, но расширение сценариев опирается на дальнейшую реализацию API.

**Что должно появиться:**
- Инициализирован проект бота (`bot/`, aiogram 3.x, long polling)
- Реализованы handlers для сценариев: регистрация, просмотр услуг/цен, запись на конкретную дату, просмотр записей, отмена записи, баланс бонусов
- Бот общается с backend через HTTP (не содержит бизнес-логики)

**Что можно проверить:**
- Новый пользователь регистрируется через бота и появляется в БД
- Клиент оформляет запись через бота, слот блокируется в расписании
- Клиент видит актуальный баланс бонусов и историю последних визитов

**Выходные артефакты:**
- `bot/` — handlers, config, точка входа
- `docs/tasks/tasklist-02-bot.md`, `docs/tasks/tasklist-02-integrations.md`

**Связь с tasklists:**
- [`tasklist-02-bot.md`](tasks/tasklist-02-bot.md)

### Этап 3 — LLM-консультант

**Цель:** Добавить диалоговую консультацию через LLM: бот отвечает на вопросы об услугах, ценах и слотах, опираясь на данные backend, а не на собственные «знания» модели.

**Ценность этапа:**
LLM-консультант — ключевая дифференцирующая фича продукта. Клиент перестаёт звонить администратору с типовыми вопросами. Все ответы модели строго привязаны к актуальному прайсу и расписанию — нет риска «выдумать» цену или несуществующий слот.

**Результат:** Бот отвечает на свободные вопросы клиентов об услугах и помогает с выбором записи через диалог.

**Статус:** ✅ done

**Факт реализации:** в backend добавлены `pereobuyka/llm` (OpenRouter) и оркестратор консультации с function-calling (`list_services`, `list_slots`, `create_appointment`) поверх существующих сервисов; эндпоинт `POST /api/v1/consultation/messages` возвращает ответ и безопасные `503` при сбоях провайдера. В боте вход — команда `/ask` (вопрос уходит в backend, ключи OpenRouter настраиваются в `backend/.env`).

**Активные области:** backend, integrations

**Зависимость:** Этап 2 завершён. Параллельно с этапом 4 возможна разработка LLM-модуля backend, если контракты зафиксированы.

**Что должно появиться:**
- В backend реализован LLM-модуль: системный промпт, формирование контекста (прайс, слоты, бонусы), вызов OpenRouter
- Обработка ошибок LLM: таймауты, недоступность провайдера — без утечки деталей пользователю
- Бот отправляет вопрос в backend (`/ask`) и возвращает ответ пользователю

**Что можно проверить:**
- Вопрос «сколько стоит шиномонтаж» возвращает актуальную цену из прайса, а не «придуманную»
- При недоступности LLM бот отвечает понятным fallback-сообщением
- Запрос «запишите меня» через диалог приводит к оформлению записи (function calling или интерактивный сценарий)

**Выходные артефакты:**
- `backend/src/pereobuyka/llm/` — клиент, промпты, контекст
- `docs/tasks/tasklist-03-llm.md`

**Связь с tasklists:**
- [`tasklist-03-llm.md`](tasks/tasklist-03-llm.md)

### Этап 4 — Административный веб-интерфейс

**Цель:** Дать администратору сервиса инструмент для самостоятельного управления прайсом, расписанием и записями без участия разработчика.

**Ценность этапа:**
До этого этапа любое изменение прайса или расписания требует прямой правки данных в БД. Административный интерфейс переводит операционное управление к владельцу сервиса. Это снимает операционную зависимость от разработчика и делает продукт самостоятельным.

**Результат:** Веб-панель администратора, через которую можно управлять прайсом, расписанием, просматривать записи и подтверждать визиты.

**Статус:** ⚪ planned

**Активные области:** frontend, backend

**Зависимость:** Этап 1 завершён (API-контракты готовы). Разработка frontend может идти параллельно с этапами 2–3 после фиксации контрактов backend.

**Что должно появиться:**
- Определён и зафиксирован frontend-стек (см. открытый вопрос в `docs/vision.md`)
- Инициализирован проект `web/`
- Реализованы экраны: управление услугами (прайс), управление расписанием, журнал записей, подтверждение визита, просмотр и корректировка бонусных счетов

**Что можно проверить:**
- Администратор добавляет новую услугу через веб — она сразу доступна в боте
- Администратор закрывает визит — бонусы начисляются клиенту автоматически
- Администратор блокирует день в расписании — слоты на этот день исчезают из доступных для записи

**Выходные артефакты:**
- `web/` — frontend-проект (стек TBD)
- ADR на выбор frontend-стека
- `docs/tasks/tasklist-04-admin-web.md`

**Связь с tasklists:**

### Этап 5 — Клиентский веб-интерфейс

**Цель:** Реализовать альтернативный канал для клиента через веб — те же сценарии, что и в боте, без привязки к Telegram.

**Ценность этапа:**
Клиентский веб снимает зависимость от Telegram как единственного канала. Клиент без Telegram-аккаунта или предпочитающий веб получает полноценный доступ к сервису. Кроме того, веб-канал является резервным при недоступности Telegram (см. `docs/tech/integrations.md`).

**Результат:** Веб-интерфейс клиента: консультация, просмотр прайса, запись, история визитов, баланс бонусов — идентично возможностям бота.

**Статус:** ⚪ planned

**Активные области:** frontend, backend

**Зависимость:** Этап 4 завершён (frontend-инфраструктура и стек зафиксированы). Часть компонентов переиспользуется из административного интерфейса.

**Что должно появиться:**
- Реализованы сценарии клиента в веб: регистрация/авторизация по телефону, просмотр прайса, запись, история визитов, управление записями, бонусный счёт
- Роль `client` изолирована от роли `admin` на уровне маршрутизации и прав

**Что можно проверить:**
- Клиент без Telegram регистрируется и записывается через веб
- История визитов и бонусный счёт синхронизированы между ботом и веб (единый backend)
- Клиент не видит административные разделы

**Выходные артефакты:**
- Клиентская часть `web/` (новые экраны и маршруты)
- `docs/tasks/tasklist-05-client-web.md`

**Связь с tasklists:**

### Этап 6 — Production-ready и delivery

**Цель:** Подготовить систему к реальной эксплуатации: контейнеризация, CI/CD, мониторинг, конфигурирование среды.

**Ценность этапа:**
Без этого этапа система работает только локально и требует ручного деплоя. Production-ready слой делает доставку изменений предсказуемой, снижает стоимость сопровождения и повышает надёжность в эксплуатации. Это также фиксирует операционные практики: логирование, секреты, откаты.

**Результат:** Система разворачивается автоматически; CI проверяет каждый PR; деплой в production воспроизводим и задокументирован.

**Статус:** ⚪ planned

**Активные области:** devops

**Зависимость:** Этапы 1–3 завершены для минимального деплоя бота; этапы 4–5 для полного деплоя системы. Базовую инфраструктуру (Docker, CI) можно готовить параллельно начиная с этапа 2.

**Что должно появиться:**
- `Dockerfile` и `docker-compose` для всех компонентов (backend, bot, web)
- CI-пайплайн: линтер, тесты, сборка образов
- PostgreSQL в production (миграции автоматизированы)
- Документация по деплою и конфигурированию среды

**Что можно проверить:**
- `docker-compose up` поднимает всю систему с нуля без ручных шагов
- CI падает при непройденных тестах и не допускает мёрдж
- Ротация секретов (смена токена бота, ключа LLM) не требует пересборки образов

**Выходные артефакты:**
- `Dockerfile`, `docker-compose.yml`
- CI-конфигурация (`.github/workflows/` или аналог)
- `docs/tasks/tasklist-06-devops.md`
- `README.md` с инструкцией по запуску

**Связь с tasklists:**
- [`tasklist-06-devops.md`](tasks/tasklist-06-devops.md)
