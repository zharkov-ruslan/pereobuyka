# Tasklist: Backend

## Область: backend

Backend — ядро системы «Переобуйка». Вся бизнес-логика живёт здесь: расчёт слотов, управление прайсом, записи, визиты, лояльность, LLM-консультация. Telegram-бот и веб-приложение — клиенты этого backend, они не содержат бизнес-правил. Смена или добавление канала не затрагивает ядро.

Детали архитектуры: [`docs/vision.md`](../vision.md) · Доменная модель: [`docs/data-model.md`](../data-model.md) · Интеграции: [`docs/integrations.md`](../integrations.md)

---

## Связь с plan.md

Tasklist покрывает следующие этапы [`docs/plan.md`](../plan.md):

| Этап плана | Название | Участие backend |
|------------|----------|-----------------|
| 1 | Фундамент: backend и модель данных | Основной исполнитель |
| 2 | Telegram-бот MVP | Backend как единая точка входа для бота |
| 3 | LLM-консультант | LLM-модуль полностью в backend |
| 4–5 | Веб-интерфейсы | Расширение API под нужды веб-клиентов |

### iter-01-stack — Выбор стека и ADR

**Что должно появиться:**

### iter-04-core — Доменные модели и сервисы

**Связь с этапом плана:** 1

**Цель:** Реализовать бизнес-логику системы: доменные модели, репозитории и сервисы для всех сущностей из `docs/data-model.md`.

**Ценность:** После этой итерации ядро системы работает независимо от канала. Вся логика расчётов и правил находится в одном месте и покрыта тестами — риск регрессий при подключении новых клиентов минимален.

**Что должно появиться:**
- Доменные модели: `User`, `Service`, `Schedule`, `Appointment`, `Visit`, `BonusAccount`, `BonusTransaction`, `FAQ`
- Репозитории с явными методами для каждой сущности (CRUD + специализированные запросы)
- Сервисы: управление прайсом, расписанием, расчёт слотов, создание и отмена записей, закрытие визита, начисление/списание бонусов

**Что можно проверить:**
- Расчёт длительности и стоимости записи по набору услуг корректен
- Алгоритм поиска свободных слотов учитывает рабочее расписание и существующие записи
- Начисление бонусов после подтверждения визита отражается в `BonusAccount` и `BonusTransaction`

**Задачи:**
- [ ] Реализовать доменные модели (dataclass или Pydantic) для всех сущностей
- [ ] Реализовать репозитории в `storage/` с интерфейсами, не привязанными к конкретной СУБД
- [ ] Реализовать `SlotService` — расчёт свободных окон по расписанию и занятым записям
- [ ] Реализовать `AppointmentService` — создание, отмена записи, расчёт стоимости и длительности
- [ ] Реализовать `VisitService` — закрытие визита, фиксация фактических услуг
- [ ] Реализовать `LoyaltyService` — начисление и списание бонусов, история транзакций

**Артефакты:**
- `backend/src/pereobuyka/models/` — доменные модели
- `backend/src/pereobuyka/storage/` — репозитории
- `backend/src/pereobuyka/services/` — бизнес-сервисы

### iter-05-api-impl — Реализация API

**Связь с этапом плана:** 1–2

**Цель:** Реализовать все эндпоинты по зафиксированному контракту и сделать backend пригодным для подключения бота.

**Ценность:** После этой итерации Telegram-бот и веб-клиент могут подключаться к backend и выполнять все клиентские сценарии. Backend является единственной точкой входа — бот не содержит бизнес-логики.

**Что должно появиться:**
- Реализованные роутеры для всех доменных областей: пользователи, услуги, расписание, записи, визиты, лояльность
- Обработка ошибок на уровне API: валидация, not found, конфликты слотов
- Интеграционные тесты — сквозные сценарии через HTTP-клиент

**Что можно проверить:**
- `POST /appointments` с набором услуг возвращает запись с корректными `starts_at`, `ends_at`, `total_price`
- `GET /slots?date=...&services=...` возвращает только реально свободные окна
- `POST /visits/{id}/confirm` начисляет бонусы и обновляет баланс клиента

**Задачи:**
- [ ] Реализовать роутеры: `/users`, `/services`, `/schedule`, `/appointments`, `/visits`, `/loyalty`
- [ ] Подключить сервисы из iter-04 к роутерам через dependency injection
- [ ] Реализовать централизованную обработку ошибок (422, 404, 409)
- [ ] Написать интеграционные тесты для ключевых сценариев (запись, слоты, бонусы)
- [ ] Проверить соответствие реализации контракту из iter-03

**Артефакты:**
- `backend/src/pereobuyka/` — роутеры, middleware, обработка ошибок
- `tests/integration/` — интеграционные тесты
- `docs/tasks/impl/backend/iter-05-api-impl/plan.md`

**Зависимости:** iter-04-core завершён; database-миграции применены (см. `tasklist-database.md`).

**Статус:** ⚪ planned

### iter-06-llm — LLM-модуль в backend

**Связь с этапом плана:** 3

**Цель:** Реализовать LLM-консультацию в backend: формирование контекста из данных сервисов, вызов OpenRouter, возврат ответа клиенту.

**Ценность:** Консультационный диалог становится управляемым — модель отвечает строго на основе актуального прайса и расписания. Смена провайдера или модели — правка конфига без изменений кода. Бот не хранит никакой LLM-логики.

**Что должно появиться:**
- LLM-клиент в `llm/` — обёртка над OpenRouter через `openai` SDK
- Сервис формирования контекста: прайс, свободные слоты, бонусный баланс клиента, FAQ
- Системный промпт зафиксирован и не зашит в код — вынесен в конфиг или файл
- Эндпоинт `POST /consult` — принимает вопрос и `user_id`, возвращает ответ модели
- Fallback при таймауте или недоступности LLM

**Что можно проверить:**
- Вопрос о цене услуги возвращает данные из актуального прайса, а не «придуманные»
- При недоступности OpenRouter API возвращает понятную ошибку (не 500 с трассировкой)
- Смена модели в `.env` без пересборки меняет используемую модель

**Задачи:**
- [ ] Реализовать LLM-клиент (`llm/client.py`) — вызов OpenRouter, обработка таймаута
- [ ] Реализовать `ContextBuilder` — сборка контекста из сервисов (прайс, слоты, бонусы, FAQ)
- [ ] Вынести системный промпт в конфигурируемый файл/шаблон
- [ ] Реализовать эндпоинт `POST /consult`
- [ ] Реализовать fallback-ответ при ошибке провайдера
- [ ] Написать тест: корректность контекста при заданном состоянии прайса и расписания

**Артефакты:**
- `backend/src/pereobuyka/llm/` — client, context builder, prompts
- `docs/tasks/impl/backend/iter-06-llm/plan.md`

**Зависимости:** iter-05-api-impl завершён (сервисы доступны для построения контекста).

**Статус:** ⚪ planned

## Качество и инженерные практики

Не выделяется в отдельную итерацию, но соблюдается на всём протяжении разработки backend.

**Тесты:**
- Unit-тесты сервисов (слоты, бонусы, расчёт записи) — в iter-04
- Интеграционные тесты API через HTTP-клиент — в iter-05
- Тест корректности LLM-контекста — в iter-06

**Линтер и форматирование:**
- `ruff` — линтер и форматтер; конфиг в `pyproject.toml`
- `make lint` — обязательный шаг перед мёржем

**Контракты:**
- Изменение публичного API — только через явную правку схем в iter-03-контракте
- Обратно несовместимые изменения — фиксируются как отдельное решение (ADR или комментарий в PR)

**Логирование:**
- `logging` stdlib; уровень из env
- Не логировать: токены, ключи, `telegram_id` без необходимости
- Ошибки интеграций — с диагностичным контекстом, без тела секретов

**Наблюдаемость:**
- `GET /health` — базовая проверка доступности
- Структурированные логи с уровнем и контекстом запроса (request_id при наличии)
