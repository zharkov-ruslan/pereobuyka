## 1. Краткие функциональные требования по экранам

### Экран «Панель администратора»

| Блок UI | Требование |
|--------|------------|
| KPI «сегодня» | Записей всего; визитов; отмен; разбивка записей по источнику (LLM / бот / админ); число вопросов боту за последние 7 дней. |
| Сетка недели | Колонки — дни текущей недели (пн–вс), строки — слоты времени; ячейка: запись / подтверждённый визит / отмена (серый); тултип — услуги, сумма; клик по занятой ячейке — модалка записи. |
| Модалка записи | Клиент, телефон; услуги с временем и ценой; итог время/сумма до скидки и со скидкой; будущие бонусы при подтверждении; кнопки отмены записи и подтверждения визита. |
| После «подтвердить визит» | Модалка оценки клиента админом: 5 звёзд + комментарий. |
| Вкладка «статистика недели» | Топ-3 услуги; один график записей/визитов/отмен по дням; график дохода по дням; график записей по источнику по дням. |

### Экран «Клиенты сервиса»

| Блок UI | Требование |
|--------|------------|
| Таблица | Все клиенты по алфавиту; колонки: визиты, сумма трат, бонусы, рейтинг (1–5, два знака). |
| Карточка | Имя, телефон; иконка Telegram → открыть чат (если есть связь); рейтинг. |
| Вкладка записей | Предстоящие + отменённые (отмены скрыты по умолчанию); услуги, цены, длительности, суммы; подтверждение/отмена; при подтверждении — модалка оценки клиента; **редактирование записи**: +/- услуги, скидка, пересчёт бонусов и суммы. |
| Вкладка визитов | Подтверждённые визиты; услуги, суммы, скидка %, оплата, бонусы, оценка и комментарий сервиса; **редактирование визита** (услуги, суммы, бонусы). |

### Экран «Личный кабинет клиента»

| Блок UI | Требование |
|--------|------------|
| Шапка | Приветствие по имени; «Запись в сервис»; переход в Telegram-бот. |
| Визард записи | На той же странице; выбор услуг; длительность, сумма, бонусы; ссылка «На главную»; **черновик в LocalStorage** до подтверждения, сброс после успешной записи. |
| Записи | Список с деталями; отмена; отменённые скрыты по умолчанию. |
| Визиты | История; **оценка сервиса** 1–5 + комментарий по каждому визиту. |

### Глобальный чат с AI

| Блок UI | Требование |
|--------|------------|
| Оболочка | Плавающая кнопка на любом экране; панель поверх контента; не отдельный роут. |
| Содержимое | История сообщений (прокрутка); ввод и отправка; ответ с задержкой. |

### Вход

| Блок UI | Требование |
|--------|------------|
| Простая форма | Без «тяжёлой» OAuth на первом шаге; идентификация через **Telegram username** (как в UI-требованиях). |

## 2. Решение по входу (веб)

**Конфликт:** [`ui-requirements.md`](../../../ui/ui-requirements.md) описывает ввод **telegram username**; действующий [`POST /api/v1/auth/telegram`](../../../tech/api/api-contracts.md#post-auth-telegram) принимает **`telegram_id`** (число) и ориентирован на бота — см. интеграции и тесты backend.

**Решение для MVP веба:**

1. **Не менять** `POST /api/v1/auth/telegram` — сохранить совместимость с ботом и существующими тестами.
2. Добавить в **iter-fe-01** отдельный эндпоинт **`POST /api/v1/auth/web`** (без Bearer):
   - Тело (черновик): `telegram_username` (строка, формат `@name` или `name` по соглашению), опционально `name`, `phone`.
   - Поведение: upsert пользователя с `role: client`, `source: web`, выдача JWT как у telegram-auth; **`telegram_id`** может оставаться `null` до явной привязки аккаунта.
3. **Дисклеймер безопасности:** ввод username без доказательства владения — это **демо/MVP**; последующая итерация — Telegram Login Widget / deep-link с верификацией — отдельная задача.

**Админ-панель:** по [`tasklist-frontend`](../../../tasklist-frontend.md) выдача JWT администратора остаётся на стороне backend/seed (**iter-fe-01**); веб админа не использует `auth/web` клиента.

## 3. Карта «экран → данные → API»

Легенда: **есть** — покрыто текущим контрактом; **дыра** — нужны новые/расширенные эндпоинты или поля в **iter-fe-01**.

### 3.1 Панель администратора

| Данные UI | Текущее API | Статус |
|-----------|-------------|--------|
| KPI на сегодня (записи, визиты, отмены) | — | **дыра** |
| KPI по источнику записи (LLM / бот / админ) | — | **дыра** |
| Вопросы к боту за 7 дней | — | **дыра** (нужны логи консультаций / событий бота) |
| Сетка слотов на неделю с состоянием ячейки | — | **дыра** |
| Журнал записей с пользователем, фильтры по датам | `GET /api/v1/admin/appointments` | частично (нет источника записи и агрегата «по слоту» без клиентских запросов) |
| Отмена записи | Нет PATCH админом в контракте | **дыра** |
| Подтверждение визита | `POST /api/v1/admin/visits` | **есть** |
| Детали записи в модалке (услуги, суммы, скидка, бонусы) | ответы `Appointment` сейчас без детализации скидки / источника | **дыра** (расширить схемы) |
| Оценка клиента администратором после визита | — | **дыра** |
| Недельная статистика (графики, топ услуг) | — | **дыра** |

### 3.2 Клиенты сервиса

| Данные UI | Текущее API | Статус |
|-----------|-------------|--------|
| Список клиентов с метриками | — | **дыра** |
| Карточка клиента, записи, визиты | Потребуются выборки по `user_id` | **дыра** (`GET .../users` или `/admin/clients/...`) |
| Редактирование записи админом | `PATCH` только у клиента на своей записи | **дыра** |
| Редактирование визита админом | только `POST` создания визита | **дыра** (`PATCH` визита или пересоздание по правилам) |
| Рейтинг клиента (агрегат) | нет в `User` / визитах | **дыра** |
| Ссылка в Telegram по клику | нужен **username** или deeplink; в модели сейчас в основном `telegram_id` | **дыра** |

### 3.3 Личный кабинет клиента

| Данные UI | Текущее API | Статус |
|-----------|-------------|--------|
| Профиль / приветствие | `GET /api/v1/me` | **есть** |
| Каталог услуг | `GET /api/v1/services` | **есть** |
| Слоты | `GET /api/v1/slots` | **есть** |
| Правила лояльности | `GET /api/v1/loyalty/rules` | **есть** |

| Список записей, отмена | `GET/PATCH /api/v1/me/appointments` | **есть** |
| Список визитов | `GET /api/v1/me/visits` | частично |
| Оценка сервиса по визиту | — | **дыра** |

### 3.4 Глобальный чат

| Данные UI | Текущее API | Статус |
|-----------|-------------|--------|
| Отправка сообщения | `POST /api/v1/consultation/messages` | **есть** |
| История переписки в панели | только ответ на один запрос | **дыра** (`GET` истории по сессии/пользователю) |

### 3.5 Вход

| Данные UI | Текущее API | Статус |
|-----------|-------------|--------|
| Вход по telegram username | `POST /auth/telegram` с `telegram_id` | **несовместимо** → решение §2 (`POST /auth/web`) |

## 4. Черновик новых и расширенных эндпоинтов (для iter-fe-01)

Ниже — целевая форма контрактов; финальные пути/имена полей — при оформлении по **api-contract-custom** и синхронизации с OpenAPI.

### Auth (без Bearer)

| Метод | Путь | Роль | Назначение |
|-------|------|------|------------|
| POST | `/api/v1/auth/web` | публично | Вход клиента в веб по `telegram_username` (+ опц. имя/телефон), JWT |

### Admin (Bearer, admin)

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/api/v1/admin/dashboard/today` | KPI на сегодня + счётчик вопросов LLM/боту за 7 дней (если объединяем метрики в один ответ) |
| GET | `/api/v1/admin/dashboard/week-grid` | Query: `week_start=date`. Сетка: для каждого слота и дня — список событий `{ state, appointment_id?, visit_id?, total_price, service_summaries[] }` |
| GET | `/api/v1/admin/analytics/week` | Query: `week_start`. Топ услуг, ряды для графиков (записи/визиты/отмены, доход, записи по источникам по дням) |
| GET | `/api/v1/admin/clients` | Пагинация; сортировка по имени; агрегаты: visits_count, total_spent, bonus_balance, rating_avg |
| GET | `/api/v1/admin/clients/{user_id}` | Карточка + вложенные списки или ссылки на отдельные ресурсы |
| GET | `/api/v1/admin/users/{user_id}/appointments` | Записи клиента (фильтры статус/даты) |
| PATCH | `/api/v1/admin/appointments/{appointment_id}` | Изменение услуг/скидки/статуса админом |
| PATCH | `/api/v1/admin/visits/{visit_id}` | Корректировка фактических услуг/сумм/бонусов |
| POST | `/api/v1/admin/visits/{visit_id}/client-rating` | Оценка клиента администратором (stars 1–5, comment); идемпотентность или upsert по визиту |

*Альтернатива меньшему числу путей:* один `GET /api/v1/admin/dashboard?date=...` с флагами `include=week_grid,analytics` — допустимо, если ответ не раздувается без нужды (см. api-design-principles: версионирование и читаемость ресурсов).

### Client (Bearer, client)

| Метод | Путь | Назначение |
|-------|------|------------|
| POST | `/api/v1/me/visits/{visit_id}/service-rating` | Оценка сервиса клиентом (stars, optional comment) |

### Consultation (Bearer, client)

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/api/v1/consultation/messages` | Query: `limit`, `offset` или курсор — история сообщений пользователя для виджета |

## 5. Черновик расширений схем (поля)

Перенос в [`data-model.md`](../../../tech/data-model.md) и Pydantic — в **iter-fe-01**.

| Сущность / объект | Новые / уточняемые поля |
|-------------------|-------------------------|
| `User` | `telegram_username` (nullable, уникальность при not null); при необходимости индекс для поиска при `auth/web` |
| `Appointment` | `source` enum: `llm` \| `telegram_bot` \| `web` \| `admin`; `discount_percent` или денежные `subtotal`, `discount_amount`, `total_price`; опционально `created_by_user_id` для админских записей |
| `Visit` | связь с оценками или встроенные `client_rating_stars`, `client_rating_comment`, `service_rating_stars`, `service_rating_comment` — либо отдельные таблицы `visit_ratings` / `service_feedback` |
| Лог консультаций | таблица сообщений (user_id, role user/assistant, text, created_at, request_id) для истории чата и метрик «вопросов за неделю» |

## 7. Отклонения и допущения

- **Один ресурс «слот» в сетке:** фактическая гранулярность (шаг 15/30 мин) задаётся backend при генерации сетки; UI следует ответу API.
- **Метрика «вопросов боту за неделю»:** может включать только LLM-консультации из backend; сообщения только в Telegram без записи в ядре — уточнить в интеграциях позже.
- **Проверка владения Telegram** для production — вне scope iter-fe-00.
