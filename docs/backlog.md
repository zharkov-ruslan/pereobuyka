## Frontend (веб)

- **Админ-панель — статистика недели до уровня спеки:** в [`docs/ui/ui-requirements.md`](ui/ui-requirements.md) (экран 1) заложены отдельная **вкладка** и **графики** (записи/визиты/отмены на одном графике, доход по дням, записи по каналам LLM/бот/админ по дням). Сейчас на панели — боковая колонка на той же странице с компактными индикаторами без полноценных графиков и без отдельного таба. Довести UI/API при необходимости.
- **ЛК клиента — история движений бонусов:** по [`docs/tech/user_scenarios.md`](tech/user_scenarios.md) клиент видит баланс и **историю** операций (`BonusTransaction`). Сейчас в веб-кабинете отображаются баланс и агрегаты по визитам, без ленты транзакций; при отсутствии read-only API — добавить контракт и экран/блок.
- **Главная `web` (`app/page.tsx`):** устаревший маркетинговый текст («следующие итерации» и т.п.) не отражает фактический scope; заменить на актуальное описание или редирект после входа без «заглушки-дорожной карты».
- При росте экранов админки — часть загрузок ближе к **Server Components** / границы **Suspense**, когда появятся серверные данные с backend.
- Рассмотреть **`next/dynamic`** для редких тяжёлых виджетов после **bundle-анализа**.
- **Сессия в UI:** в `ChatWidget` сейчас `loadAuthSession()` на рендере; при артефактах (лишние перезапросы, устаревший токен в UI) вынести в **контекст** или подписку на обновление после логина.

## Backend: архитектура (Onion / DDD)

- Вынести сценарии «подтверждение визита», «корректировка бонусов», «отмена записи» в **use case**-классы с одним публичным методом (`execute`), оставить в `storage` только адаптеры к БД.
- Ввести **интерфейсы репозиториев** в доменном слое и реализации в infrastructure — сейчас репозитории завязаны на SQLAlchemy напрямую.
- Домен без зависимостей от FastAPI/Pydantic в ядре (сущности и правила — отдельно от HTTP-схем).

---

## Backend: качество, CI, тесты

- Подключить **`mypy src`** (или `ty`) в CI: в `pyproject.toml` уже есть базовая конфигурация, нужна регулярная проверка.
- При росте интеграционных тестов — маркер **`@pytest.mark.integration`** и цель Makefile вроде `pytest -m "not integration"` для быстрых локальных прогонов.
- Узкие тесты на **негативные пути** API: 401 без токена, 403 для админских маршрутов с неверным `ADMIN_API_TOKEN`, 409 при конфликтах домена.
- Для сценариев с временем при необходимости — **freezegun** или фикстуры с фиксированным `datetime`.

## API и контракты

- Держать **матрицу кодов ошибок** в sync с [`docs/tech/api/errors.md`](tech/api/errors.md) и фактическими `HTTPException` в коде.
- В OpenAPI явно описать **две схемы авторизации**: клиентский Bearer (`BOT_SECRET` и пользовательский токен после auth) и админский (`ADMIN_API_TOKEN`).
