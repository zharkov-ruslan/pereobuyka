# api-contract-custom — правила оформления api-contracts.md

Скилл описывает соглашения, обязательные при создании и редактировании файла `docs/tech/api/api-contracts.md` в проекте «Переобуйка».

## Когда использовать

- При добавлении нового эндпоинта в `api-contracts.md`.
- При ревью или рефакторинге существующих разделов.
- При синхронизации `api-contracts.md` с `openapi.yaml`.

---

## 1. Структура документа

Файл состоит из секций по группам доступа. Заголовки секций обязательно указывают роль и наличие Bearer:

| Формат заголовка | Когда использовать |
|---|---|
| `## ИмяГруппы (без Bearer)` | Публичные и аутентификационные эндпоинты |
| `## ИмяГруппы (Bearer, роль client)` | Эндпоинты только для клиентов |
| `## ИмяГруппы (Bearer, роль admin)` | Эндпоинты только для администраторов |

Все группы в документе:

```
## Auth (без Bearer)
## Public (без Bearer)
## Client (Bearer, роль client)
## Consultation (Bearer, роль client)
## Admin (Bearer, роль admin)
```

---

## 2. Сводная таблица эндпоинтов

Каждая группа открывается сводной таблицей с колонками:

```
| Метод | Путь | Описание | Успех |
```

Пути в таблице — чистые, без query-параметров. Все параметры и заголовки описаны в детальных подразделах ниже.

### Правила колонки «Успех»

- Один код: `200` / `201` / `204`.
- При upsert, где сервер возвращает разные коды: `200 / 201`.

---

## 3. Детальный раздел эндпоинта

Каждый эндпоинт описывается в отдельном подразделе после таблицы. Структура:

```markdown
<span id="уникальный-якорь"></span>

### МЕТОД `/api/v1/путь`

**Описание:** <однострочное описание>

**Заголовки:**

| Заголовок | Значение | Обязательный |
|---|---|---|
| `Authorization` | `Bearer <JWT>` | да |
| `Content-Type` | `application/json` | да |

**Параметры:** —                   ← если параметров нет

**Параметры:**                     ← если параметры есть (path и/или query)

| Параметр | Расположение | Тип | Обязательный | Описание |
|---|---|---|---|---|
| `param_name` | path | UUID | да | Описание. |
| `filter` | query | string | нет | Описание. |

**Запрос:** строка или http-блок

**Ответ `200`/`201`/`204`:**

{ ... }

**Ответ `4XX`** (описание):

{ ... }
```

### Правила блоков параметров

Блоки **Заголовки** и **Параметры** — обязательны в каждом детальном разделе, даже если значение `—`.

**Заголовки** — `**Заголовки:** —` если заголовков нет. Если есть — таблица:

```markdown
**Заголовки:**

| Заголовок | Значение | Обязательный |
|---|---|---|
| `Authorization` | `Bearer <JWT>` | да |
| `Content-Type` | `application/json` | да |
```

Типовые наборы:
- Публичный endpoint (без Bearer): `**Заголовки:** —`
- Только Bearer (GET с токеном): таблица с одной строкой `Authorization`
- Bearer + JSON-тело (POST/PATCH): таблица с двумя строками
- Только Content-Type (POST без Bearer): таблица с одной строкой `Content-Type`

**Параметры** — одна строка `**Параметры:** —` если параметров нет. Если параметры есть (path и/или query) — таблица с колонками:

```markdown
| Параметр | Расположение | Тип | Обязательный | Описание |
|---|---|---|---|---|
| `service_id` | path | UUID | да | UUID услуги. |
| `is_active` | query | bool | нет | Фильтр по активности. |
```

Правила:
- Значения колонки **Расположение**: `path` или `query`.
- Значения колонки **Обязательный**: `да` или `нет`.
- Path-параметры всегда `да` (обязательный).
- В колонке **Параметр** имя без фигурных скобок: `service_id`, не `{service_id}`.
- Если у эндпоинта есть и path-, и query-параметры — они в одной таблице.
- Body-атрибуты (поля тела запроса) в таблицу не включаются.

**Запрос** — после блоков Заголовки/Параметры: строка для простых случаев, ` ```http ` -блок для сложных.

## 3. Детальный раздел эндпоинта

Каждый эндпоинт описывается в отдельном подразделе после таблицы. Структура:

```markdown
<span id="уникальный-якорь"></span>

### МЕТОД `/api/v1/путь`

**Описание:** <однострочное описание из таблицы>

**Запрос:** <строка запроса или блок ```http с заголовками>

**Ответ `200`** / **`201`**:

```json
{ ... пример тела ... }
```

**Ответ `4XX`** (описание ошибки):

```json
{ ... пример тела ошибки ... }
```
```

### Якоря

Якоря задаются явно через `<span id="...">` — перед заголовком подраздела. Это гарантирует работу ссылок в любом Markdown-просмотрщике независимо от генерации slug.

Конвенция именования якоря: `{метод}-{ресурс-через-дефис}`, например:
- `get-me`, `post-appointments`, `patch-me-appointment`
- `get-admin-services-list`, `post-admin-services`
- `get-admin-schedule-rules`, `post-admin-schedule-exceptions`

---

## 4. Именование путей

- Коллекции: множественное число (`/services`, `/appointments`, `/visits`).
- Вложенные ресурсы одного уровня: через дефис (`/bonus-account`, `/bonus-transactions`), не через слеш (`/bonus/account` — неверно).
- Один экземпляр ресурса: `/{resource_id}` в пути.
- Путь к тому же ресурсу должен быть единообразен для клиентских и административных эндпоинтов:
  - Клиент: `GET /api/v1/me/bonus-account`
  - Администратор: `GET /api/v1/admin/users/{user_id}/bonus-account`

---

## 5. Коды ответов

| Ситуация | Код |
|---|---|
| Успешный GET или PATCH | `200` |
| Создание нового ресурса (POST) | `201` |
| Успешное удаление без тела | `204` |
| Upsert, возвращающий разные коды | `200` (существует) / `201` (создан) |
| Stateless-диалог (ответ LLM, тело не сохраняется) | `200` |
| Невалидные данные запроса | `422` |
| Конфликт состояния ресурса | `409` |
| Не найдено | `404` |
| Не авторизован | `401` |
| Нет прав | `403` |

## 6. Примеры ошибок

Для каждого эндпоинта, где возможны ошибки `409` или `422`, добавить блок с примером. Структура тела ошибки — см. `docs/tech/api/errors.md`:

```json
{
  "error": "код_ошибки",
  "message": "Человекочитаемое описание",
  "details": { ... }
}
```

Типичные примеры:

**422 — невалидные данные:**
```json
{
  "error": "validation_error",
  "message": "Request validation failed",
  "details": [{"field": "starts_at", "message": "field required"}]
}
```

**409 — конфликт:**
```json
{
  "error": "slot_conflict",
  "message": "Requested time slot is not available",
  "details": {"starts_at": "2026-04-20T10:00:00+03:00"}
}
```

## 7. Синхронизация с openapi.yaml

`api-contracts.md` — человекочитаемое дополнение к `openapi.yaml`. При изменении контракта оба файла обновляются одновременно:

| Изменение | Файлы |
|---|---|
| Переименование пути | `openapi.yaml` (ключ пути) + `api-contracts.md` (таблица, заголовки, тексты запросов) |
| Добавление/изменение параметров | `openapi.yaml` (parameters) + `api-contracts.md` (пример запроса, описание) |
| Новые коды ответов | `openapi.yaml` (responses) + `api-contracts.md` (таблица «Успех», блок ответа) |
| Добавление эндпоинта | Оба файла |

---

## 8. Эндпоинты без пагинации — обязательная документация

Коллекции без пагинации (ограниченные по природе или по фильтру) должны явно документироваться как таковые. Если `date_from`/`date_to` ограничивают выборку — они обязательны (`required: true` в openapi.yaml).

Текущие коллекции без пагинации:

| Путь | Ограничение |
|---|---|
| `GET /api/v1/services` | Каталог мал по природе предметной области |
| `GET /api/v1/admin/services` | То же |
| `GET /api/v1/admin/schedule/rules` | Ограничен обязательными `date_from`/`date_to` |
| `GET /api/v1/admin/schedule/exceptions` | Ограничен обязательными `date_from`/`date_to` |
| `GET /api/v1/slots` | Ограничен обязательными `date_from`/`date_to` |
